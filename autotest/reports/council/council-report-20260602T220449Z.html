Web research → PB verification

LLM Council verdict · 2026-06-02 · MediaHub Capability 3 (deterministic-engine boundary)
The decision How should the new bounded deep-research loop relate to personal-best (PB) baseline discovery — given a deterministic PB-discovery flow already exists — without the model ever authoring a baseline time used in the deterministic “is this a PB?” decision? Options: A don’t connect · B URL-broker fallback · C converge the two stacks.

Peer review was unanimous: the First Principles take was rated strongest by all five reviewers.

Chairman verdict

Where the council agrees

  • Option B (URL-broker fallback), not A or C. The deterministic flow already supplies baselines and already rides SearXNG; the only real gap is disambiguation for hard-to-find swimmers — what the model is good at. Model proposes where to look; the deterministic parser still extracts the number.
  • Enforce the boundary structurally: the fallback returns only list[str] URLs — a leaked time should be a type error, not a code-review note.
  • Off by default, hard-capped per run (a run-shared counter, not per-swimmer) so one big meet can’t trigger dozens of LLM loops.
  • Fail toward false negatives — a wrong PB is catastrophic for a small club.
  • Converge the two stacks later, if ever — don’t refactor a working deterministic path while adding behaviour.

Where the council clashes

  • Trigger: fire only on a true dead end (zero usable parsed PBs), never on “low confidence” — low-confidence is exactly when you must stay deterministic. (Contrarian wins over the confidence-threshold idea.)
  • Converge now? Only the Expansionist; everyone else says premature — converging before the poisoning/identity surface is hardened turns the moat into a liability. Defer.

Blind spots peer review caught

  • Identity collision (top risk): a URL can parse cleanly yet be a different same-named swimmer. Gate: parsed name must fuzzy-match the target before any time is accepted.
  • Untrusted-domain bytes: a model-proposed domain’s page must clear an authority/trust floor before it is fetched/parsed.
  • Ships dark: instrument every fire (swimmer, URLs tried, parsed/rejected).
  • No write-back = no compounding: promote a gated-success domain into the trust ledger so it doesn’t fire forever — but only after the gates pass.
  • Stale cache as false confidence — partially mitigated by the 7-day TTL; follow-up, not a blocker.
The one thing to do first: lock the two invariants — the fallback fires only when the deterministic pass returns zero usable parsed PBs, and it returns only list[str] candidate URLs — with a regression test proving the deterministic path never calls the fallback. Every other gate hangs off these safely.

Full advisor responses

The Contrarian

Option B is correct, but the most dangerous assumption is buried: the fallback trigger is undefined and that’s where the poison enters. “Fails how?” Zero rows parsed is defensible; “low confidence” hands the model a foothold to re-run exactly when you need determinism. The cap is understated — dozens of swimmers × per-swimmer fallback × multiple cycles hits Gemini free-tier limits mid-meet; without a hard ceiling (~2/run) a 40-obscure-swimmer meet blows the budget and silently leaves half the pack with no baseline. The gate is fragile: who validates the URL is a results page before fetching? A poisoned page with valid-looking HTML and fabricated times passes straight to the parser, which trusts the byte stream; a new model-proposed domain starts at zero trust — fetching it at all is the exposure. Biggest blind spot: identical swimmer names are exactly where hallucinated URLs are likeliest.

The First Principles Thinker

Strip the framing: the tension isn’t AI-vs-deterministic, it’s query-construction quality vs parsing correctness — separable, and pb_discovery already separates them; the only gap is query construction for ambiguous swimmers. Option B, narrow: one function (_build_candidate_urls()); deep_research’s only permitted output is a list of URLs (passing a time fails the type contract). Conflict rule: external baselines only RAISE the bar — reject an external time faster than club-recorded. Cap: fallback only if zero parse-confidence candidates after the normal SearXNG pass; 5/run, configurable. Converge LATER. Biggest blind spot: ledger poisoning via a model-proposed URL that parses correctly but is the WRONG same-named swimmer — the gate must require the parsed name to fuzzy-match the target before accepting a time.

The Expansionist

Option C, and the council undervalues what’s being built. The trust ledger is a swimmer-identity graph — a compounding athlete-intelligence asset (longitudinal tracking, career narratives, spotlights, sellable coach insights). Gate via distinct types: deep_research returns CandidateURLs, the parser returns ParsedBaseline — can’t be conflated. Cost: trigger on parse-confidence < ~0.7 AND cache miss; cap 2/meet; high-trust sources skip deep_research. Converge now — two parallel stacks drift. Option A throws away the disambiguation intelligence a fixed query-builder always gets wrong on common names. The moat is the ledger.

The Outsider

Option B, almost self-evident; don’t touch a working system. B adds a fallback disambiguation layer; the AI only reasons about which page, never the number. Naming danger: “agent”/“PB finder” invites trusting prose — call it “page finder.” A wrong PB is catastrophic for a small club — prefer false negatives over false positives. Option A is fine if the existing flow catches 95%+; ship A, measure the real failure rate, build B only when users hit real walls.

The Executor

Option B. Add _deep_research_fallback_urls() in pb_discovery/discover.py, called from discover_swimmer_pbs after the ranked-URL loop exhausts with zero usable PBs (if best_source is None). URLs from ResearchResult.sources (flat list), prefer authority_sources, max 2, into the existing parser; the model never touches times. Hard cap max_deep_fallbacks=0 (off by default), env MEDIAHUB_PB_DEEP_FALLBACK_LIMIT, run-shared counter via RunCache. Tests: the model’s answer is never written to a PBRow; the deterministic path never calls deep_research. Defer C, ledger updates, UI.
