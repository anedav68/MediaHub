Chairman verdict

Agree

  • Per-job atomic claim, not leader election or file lock. The claim is the mutex; there's no "leader" to reason about, and worker recycling becomes a non-event.
  • The primitive: INSERT … ON CONFLICT(task_id, fire_time_utc) DO NOTHING — one atomic write, SQLite serialises writers, the loser skips. Not a read-modify-write UPDATE.
  • Reframe: not impossible "exactly-once delivery" but idempotent execution keyed by (task, scheduled slot) — the outcome happens once.
  • fire_time = canonical scheduled slot in UTC (timezone-aware croniter), never wall-clock, so every worker computes the same dedup key.
  • Execute off the tick thread (existing threading.Thread model); WAL + busy_timeout; claim txn = one insert.

Clash → resolved

  • Recovering a dead worker's running row: a TTL _reap_stale that auto-resets it is the double-post catastrophe (any TTL that recovers a real crash also re-fires a job whose effect already happened). Resolution: never auto-reset. Recover deterministically; failed/interrupted are terminal, human-reviewed.
  • Tick mechanism: a daemon tick in every worker (the claim dedups, so multiple tickers are safe) + startup catch-up — covers the brief "both recycling" gap.

Blind spots caught (all folded in)

  • SIGTERM → explicit completion. A recycling worker marks its in-flight runs interrupted before it dies — closes the orphaned-row gap at £0, removes the need for TTL reset.
  • Claim ≠ completion. done is written only after the work (Phase 2: confirmed external success), in try/finally that records failed explicitly.
  • DST/timezone: store + dedup on UTC; a naive-local slot double-fires across a DST boundary.
  • Phase-2 idempotency may not exist on the target platform — the completion-after-success discipline + human approval is the only guard.
The one thing to do first: nail the claim primitive + its dedup key — a single INSERT … ON CONFLICT(task_id, fire_time_utc) DO NOTHING that reports whether this worker won — with the regression test that 8 concurrent threads (2 workers × 4) racing the same slot yield exactly one winner. Everything else hangs off that.

Full advisor responses

First Principles Thinker

The question is framed wrong — "exactly-once delivery" is unsolvable; you need idempotent execution keyed by (task_id, fire_time). One row per (task_id, fire_time) via INSERT OR IGNORE on a UNIQUE constraint; every worker ticks, first insert wins, others skip. No leader/lock/TTL. Recycling is a non-problem. Catch-up scans ~4h on startup. Phase-2: this is exactly-once claim not effect — add completed_at written after the external success, try/finally, mark failed explicitly.

Executor

Schema scheduled_tasks + scheduled_runs UNIQUE(task_id,fire_time); claim = INSERT, IntegrityError = lost; lock-free. fire_time = canonical croniter slot. Tick every worker 45s; spawn a daemon thread to run; WAL + busy_timeout=5000. 8-thread race test → one winner. Phase-2: run_id dedup before the external HTTP call.

Contrarian

None give exactly-once in Phase 2. SIGTERM on recycle kills the mid-API-call daemon thread → row stuck running → TTL takeover resets → re-posts. The TTL reset is the catastrophe. Fix: separate claim from completion; completed only after a committed row AND a confirmed success; never auto-reset; explicit failed. Phase-2 idempotency must live on the destination platform — the ledger can be lost, the post can't be unposted.

Outsider

B is simplest to reason about: no "leader" — every copy claims atomically and runs only if it won. Optimistic locking; restarts irrelevant. Leader-election foot-gun: the leader dies mid-job and the successor can't tell if it ran. flock fragile. Name it due_jobs, states pending→claimed→done→failed; avoid leader/master/heartbeat.

Expansionist

B — the claim primitive is the same one the autonomy runner (step 6) needs; build once, reuse. The run-history ledger seeds a "your week ahead" content calendar (proactive, Buffer/Later-like). then_task chaining = approve-Sunday pipelines, any sport. Net-new: scheduled re-ingest against a live results-feed URL — the zero-human flywheel.
