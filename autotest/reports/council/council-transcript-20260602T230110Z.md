# Council transcript

## Framed question
DECISION: How should MediaHub run an in-process job SCHEDULER so each scheduled job fires EXACTLY ONCE, given **2 Gunicorn workers** (+4 threads each) that also RECYCLE every ~200 requests (`--max-requests 200`), with **NO new infrastructure** (¬£0: SQLite + threads only; no Redis/Celery/external cron)?

CONTEXT: Flask monolith SaaS on Render. Synchronous; background work uses daemon `threading.Thread`; cross-worker state lives in the shared SQLite `data.db` under DATA_DIR. GOAL (report Cap 5 / Section 6 step 4): a `ScheduledTask` model (once/daily/weekly/monthly + cron via croniter) + `ScheduledTaskRun` history in data.db, executed via the existing threading model. Now: scheduled re-ingest + report/content-pack generation. FUTURE (Phase 2): scheduled, human-approved POSTING ‚Äî so exactly-once REALLY matters (a double-post to a club's socials is catastrophic). The report assumed 1 worker; reality is 2 ‚Äî hence this decision.

CANDIDATES: A) leader election (SQLite lease + heartbeat); B) per-job atomic claim in SQLite; C) file lock in /dev/shm.

## Advisor responses (de-anonymised)

### The First Principles Thinker
The question is framed wrong. "Exactly-once delivery" is an unsolvable distributed problem ‚Äî stop trying. What you need is **idempotent execution keyed by (task_id, scheduled_fire_time)**. Minimal primitive: one row per (task_id, fire_time) inserted via `INSERT OR IGNORE` on a UNIQUE constraint. Every worker ticks; the first to insert wins, all others get a constraint violation and skip. No leader election, no file locks, no TTL math. INSERT OR IGNORE is cleaner than UPDATE-WHERE-unclaimed (single atomic op, no read-modify-write race). Worker recycling becomes a non-problem: a recycled worker stops ticking; the next worker scans due tasks and claims. Catch-up: on startup, scan fire_times in the past N hours (e.g. 4h) with no completed run, fire once, skip older. Phase 2: INSERT OR IGNORE gives exactly-once *claim*, not exactly-once *effect* ‚Äî a thread that crashes after claiming but before posting leaves a row with no post. Add `completed_at`, written *after* the external API call succeeds, inside try/finally that marks failed runs explicitly. SQLite: `busy_timeout`; two workers racing an INSERT is exactly what SQLite handles ‚Äî one wins, one retries and hits the unique constraint. Leader election is strictly worse.

### The Executor
Option B. Schema: `scheduled_tasks(id, name, schedule_type, schedule_expr, task_type, task_params, enabled, next_fire_at)` + `scheduled_runs(id, task_id, fire_time, worker_id, started_at, finished_at, status, error, UNIQUE(task_id, fire_time))`. The claim = `INSERT` into scheduled_runs; catch `IntegrityError` ‚Üí another worker won. UNIQUE(task_id, fire_time) makes it lock-free ‚Äî no UPDATE‚Ä¶WHERE-unclaimed read-modify-write. `fire_time` = canonical slot (croniter `get_prev`), not wall-clock, so both workers compute the same string ‚Üí one claim wins. Tick loop started in every worker (the claim INSERT is the mutex), 45s interval (20s for cron-to-minute). `_reap_stale` at top of each tick: `UPDATE running‚Üífailed WHERE started_at < now-15min`. Execute in a spawned daemon thread (non-blocking tick), calling the existing pipeline entry; `finally` marks done/failed + schedules next. WAL at DB init; `busy_timeout=5000`; keep the claim txn to one INSERT (<1ms). Test: 8 threads (2 workers √ó 4) race `try_claim` on the same (task, fire_time) ‚Üí exactly one non-None. Phase 2: add a run_id dedup check in the posting code before any external HTTP call.

### The Contrarian
None of these give exactly-once in Phase 2 ‚Äî they give exactly-once-if-nothing-goes-wrong. The failure chain: a worker claims, fires a daemon thread to post, then hits its recycle window; Gunicorn sends SIGTERM; the daemon thread is mid-API-call and Python does not guarantee daemon completion on SIGTERM ‚Äî it's killed. The row is left `running`, never `completed`. Another worker's TTL takeover sees a stale claim, resets it, fires again ‚Üí **double-post**. The TTL-based reset is the catastrophe: any TTL short enough to recover a genuine crash is also short enough to double-fire on a normal mid-execution death, and job duration is unbounded (5‚Äì120s). The fix: **separate claim from completion**; mark `completed` only after a committed row AND a confirmed success response; **never auto-reset a claimed job** ‚Äî require an explicit `failed` transition. For Phase-2 posting specifically, the idempotency key must live on the **destination platform** (idempotency tokens on the social API), because your ledger can be lost but the post cannot be unposted. Option B with this correction is right; A adds coordinator complexity for no extra safety; C (flock) isn't portable across Render deploys.

### The Outsider
Option B is simplest to reason about: there is no "leader" concept ‚Äî every copy looks for unclaimed due jobs, atomically stamps its name, and runs only if the stamp succeeded ("I run only if I won the race"). Optimistic locking; restarts are irrelevant. A (lease) is a foot-gun: the leader dies mid-job after winning leadership but before completion, and the successor has no idea whether it ran ‚Äî you now need a completed flag AND a lease (most of B, plus extra parts), and "who's the leader now?" has no easy answer. C (flock) is fragile when a process dies without releasing it and breaks if the copies ever run on different machines. Dead-simple safety net against double-posting: a UNIQUE(job_type, scheduled_for) + a `ran_at` timestamp; before any external action check `ran_at IS NULL`, claim with `UPDATE ‚Ä¶ WHERE ran_at IS NULL`, confirm one row affected, else abort silently. Idempotency by content: a `job_fingerprint` = hash(type + target time); if it already exists in a completed table, skip. Naming that survives a handoff: `due_jobs` with states `pending ‚Üí claimed ‚Üí done ‚Üí failed`; avoid "leader"/"master"/"heartbeat".

### The Expansionist
B ‚Äî because the claim primitive is the **same one the autonomy runner (step 6) needs** for step execution. Build once, reuse everywhere; zero extra infra. The `ScheduledTaskRun` history ledger is the sleeper feature: timestamps, outcomes, content-pack IDs ‚Äî enough to build an operator-facing **"your week ahead"** view (scheduled / generated / pending-approval / posted). That turns MediaHub from reactive (upload‚Üíprocess‚Üíexport) into proactive (here's what's coming, approve Thursday's three posts) ‚Äî a Buffer/Later-class product, unlocking recurring-pricing logic. `then_task` chaining (ingest‚Üídetect‚Üíbrief‚Üírender‚Üíqueue-for-approval on a cadence) means a committee member does nothing but approve Sunday morning; the same chain works for any sport once the interpreter is sport-agnostic. The undervalued net-new: **scheduled re-ingest against a live results-feed URL** ‚Äî poll on a cadence, detect new swims, surface content automatically. The zero-human-effort flywheel.

Blind spot: B/D recommend `_reap_stale`/TTL reset without acknowledging it's the mechanism that turns a normal mid-execution recycle into a double-post. **All missed:** register a **SIGTERM handler** to mark in-flight rows `failed` explicitly before the worker dies ‚Äî ¬£0, SQLite-only, closes the SIGTERM-kills-daemon gap without TTL reaping.

2. Strongest: B + C's correction (B = skeleton, C = the safety discipline). 3. Strongest: B (clearest schema + UNIQUE(task_id,fire_time) + WAL/busy_timeout + reaper). 4. Strongest: B + C's correction. ## Chairman verdict

### Where the Council Agrees
- **Option B ‚Äî per-job atomic claim in SQLite ‚Äî not leader election (A) or file lock (C).** Unanimous. The claim *is* the mutex; there is no "leader" to reason about, and worker recycling becomes a non-event.
- **The atomic primitive is `INSERT ‚Ä¶ ON CONFLICT/OR IGNORE` on a `UNIQUE(task_id, fire_time)` row**, not a read-modify-write `UPDATE ‚Ä¶ WHERE unclaimed`. One atomic write; SQLite serialises writers; the loser hits the constraint and skips.
- **Reframe the goal:** not impossible "exactly-once delivery" but **idempotent execution keyed by (task_id, fire_time)** ‚Äî the *outcome* happens once.
- **`fire_time` is the canonical scheduled slot, in UTC** (computed from a timezone-aware croniter), never wall-clock ‚Äî so every worker computes the identical dedup key.
- **Execute off the tick thread** (hand to the existing `threading.Thread` model) so the tick never blocks; surface via the run-status model.
- **SQLite:** WAL + `busy_timeout`; keep the claim transaction to a single insert.

### Where the Council Clashes
- **How to recover a `running` row whose worker died (recycle/crash).** B/D's instinct ‚Äî a TTL `_reap_stale` that flips `running‚Üífailed` (or resets it) after N minutes ‚Äî is exactly what the Contrarian (and 4 of 5 reviewers) identify as the **double-post catastrophe**: any TTL short enough to recover a real crash will also re-fire a job whose external effect already happened. **Resolution: do NOT auto-reset/auto-retry a `running` row.** Recover it deterministically instead (below). `failed` is a terminal, human-reviewable state ‚Äî never an auto-retry queue for posting.
- **Daemon tick thread vs tick-on-request.** A per-worker daemon tick can have a brief "both workers recycling ‚Üí no live tick" gap; tick-on-request has no thread but won't fire during zero-traffic windows. **Resolution: a daemon tick thread in every worker (the claim dedups, so multiple tickers are safe) + startup catch-up** covers both ‚Äî the brief no-tick gap is absorbed by the next tick + catch-up.

### Blind Spots the Council Caught (fold all in)
1. **SIGTERM ‚Üí explicit completion, no TTL reaping.** Register a `SIGTERM`/`atexit` handler so a recycling worker marks its in-flight runs `interrupted`/`failed` *before* it dies. This closes the orphaned-`running`-row gap at ¬£0 and removes the need for the dangerous TTL reset.
2. **Claim ‚âÝ completion.** `claimed/running` on the atomic insert; `done` (with `completed_at`) **only after** the work (Phase 2: the confirmed external success) ‚Äî inside try/finally that records `failed` explicitly.
3. **Timezone/DST.** Store and dedup on **UTC** fire-times computed from an aware croniter; a naive-local slot double-fires (or skips) across a DST boundary.
4. **Phase-2 destination idempotency may not exist.** Some social APIs offer no idempotency token, so the `completed_at`-after-success discipline + human approval is the *only* guard ‚Äî design the data model to support a destination idempotency key where one exists, but never rely on it.
5. **Catch-up window.** On startup, fire un-completed past slots within a bounded window (~hours) once; skip older (a stale re-ingest is worse than a missed one). Posting catch-up stays operator-gated.

### The Recommendation
Build **Option B** to the First Principles spec, with the Contrarian's safety discipline and the peer-review fixes:

1. `mediahub.workflow.schedule`: `ScheduledTask` (once/daily/weekly/monthly + cron via **croniter**) + `ScheduledTaskRun` in `data.db`, with **`UNIQUE(task_id, fire_time_utc)`** as the dedup key. States: `pending ‚Üí running ‚Üí done | failed | interrupted`.
2. `mediahub.scheduler`: a daemon tick (every ~30‚Äì45s) started once per worker; each due task's canonical UTC slot is **atomically claimed** via insert-or-ignore; the winner advances `next_fire_at` in the **same transaction** and hands execution to the existing threaded pipeline; losers skip.
3. **No TTL auto-reset.** A `SIGTERM`/`atexit` handler marks in-flight runs `interrupted` on recycle; startup **catch-up** re-fires only *un-started* past slots in a bounded window. `failed`/`interrupted` are terminal/operator-reviewed.
4. `mediahub.notify` + `mediahub.notify.channels`: an independent thin push/webhook client (default OFF, operator-configured), fired on queue/approval (`CardStatus`) transitions ‚Äî no scheduler coupling.
5. ¬£0 throughout (SQLite + threads + the tiny pure-Python croniter dep); `THIRD_PARTY_LICENSES` gains **croniter (MIT)** and **ntfy (Apache-2.0/GPLv2)**.
6. Phase-2 posting is **out of scope here**; the data model just must not preclude a destination idempotency key + the human-approval gate.

The Expansionist's upside (the claim primitive reused by the autonomy runner; the run-history ledger as a future "week ahead" calendar; `then_task` chaining; scheduled live-feed re-ingest) is real and the design enables it ‚Äî but none of it is built now.

### The One Thing to Do First
Nail the **claim primitive + its dedup key**: a single `INSERT ‚Ä¶ ON CONFLICT(task_id, fire_time_utc) DO NOTHING` that returns whether this worker won, with `fire_time_utc` the canonical scheduled slot ‚Äî and the regression test that 8 concurrent threads (2 workers √ó 4) racing the same slot produce **exactly one** winner. Everything else (catch-up, SIGTERM handler, execution hand-off, the notifier) hangs off that one guarantee.
