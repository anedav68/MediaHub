# Council transcript

## Original question
How should MediaHub wire web research into personal-best (PB) verification — given the product owner approved "supply missing baselines too" — without the model ever authoring a baseline time used in the deterministic "is this a PB?" decision?

## Framed question
DECISION: How should MediaHub's NEW bounded deep-research ReAct loop (`web_research.deep_research`, Cap 3b) relate to PERSONAL-BEST (PB) baseline discovery, given that a deterministic PB-discovery flow ALREADY exists — without ever letting the model's synthesized answer become a baseline time used in the deterministic "is this a PB?" comparison?

LOCKED (judge the engineering within these): owner approved involving research in PB baselines (HOW not WHETHER); the deterministic boundary holds — a baseline TIME must come from deterministic parsing of a page (`parse_pbs_from_page`/interpreter), never the model's number (model may PROPOSE where to look, not ASSERT the time); no hardcoded source domains (authority is LEARNED via `context_engine.trust` or operator-configured); £0 (ride existing Gemini free tier + in-container SearXNG/DuckDuckGo, no new infra); honest-error, no fabricated baseline.

CRITICAL EXISTING FACTS (two research stacks now): (1) `pb_discovery/discover.py` already caches → builds queries → searches via `context_engine.research.ResearchClient` (a thin adapter over `web_research.search.WebResearcher`, so it ALREADY rides the new SearXNG backend) → ranks candidate URLs by the trust ledger → fetches top ~3 → DETERMINISTICALLY parses PB rows via `parse_pbs_from_page` → picks highest parse-confidence → `record_attempt()` → caches; consumed by `pipeline_v4.py` and `pb_bridge.py`. (2) The new `deep_research` returns a SYNTHESIZED cited TEXT answer + source URLs + authority subset; discards partial synthesis; does NOT parse times. (3) pb_discovery's path is already the "safe" shape; deep_research's strength is the messy "which page / which swimmer" disambiguation a fixed query-builder is bad at — but its prose is untrustworthy as a number.

OPTIONS: A=don't route deep_research into PB (already done deterministically); B=URL-broker (only when the deterministic flow FAILS, use deep_research to DISCOVER better candidate URLs, then hand them to the EXISTING deterministic parser; model never supplies a number); C=converge the two stacks; D=better.

Pressure-tested on: which option & why; the structural gate before a model-found URL's parsed time becomes a baseline; conflict handling (club X vs authority Y); poisoning/prompt-injection of fetched pages; per-swimmer cost (dozens per meet); converge-now vs later; biggest blind spot where PB accuracy is the core trust promise.

## Advisor responses

### The Contrarian
Option B is correct, but the proposal buries its most dangerous assumption: **the fallback trigger is undefined and that's where the poison enters.** "When pb_discovery fails" — fails how? Zero rows parsed is defensible; "low confidence" hands the model a foothold to re-run exactly when you most need determinism, not a prompt-injectable reasoning loop. The cap problem is understated: dozens of swimmers × per-swimmer fallback × multiple search+fetch cycles will hit Gemini free-tier limits mid-meet; without a hard integer ceiling (say 2 per run), one meet with 40 obscure swimmers blows the budget and leaves half the pack with no baseline — silent partial failure, worse than honest failure. The structural gate is fragile: "model proposes URL, deterministic code extracts time" is right, but who validates the URL is a results page before fetching? A poisoned page returning valid-looking HTML with fabricated times passes straight to `parse_pbs_from_page`; the parser trusts the byte stream and has no notion of source credibility. A new model-proposed domain starts at zero trust — fetching it at all is the exposure. Biggest blind spot: small clubs have swimmers with identical names; the model's "disambiguation" is exactly where hallucinated URL proposals are most likely. Trust nothing the model says about which swimmer is which.

### The First Principles Thinker
Strip the framing first. The tension isn't "AI vs deterministic" — it's **query-construction quality vs parsing correctness**. These are separable; pb_discovery already separates them; the bug is only query construction for ambiguous swimmers. What are we solving? pb_discovery fails when the fixed query-builder produces bad candidate URLs; deep_research is better at disambiguation *before* URL selection. That's the only gap; everything downstream works. Option B is correct and the scope is narrow: the integration point is one function (`_build_candidate_urls()`); deep_research's only permitted output into that path is a list of URLs — not times, confidence, or prose. Build the interface so the wrong thing is impossible: returns `List[str]`; passing a time fails the type contract. Conflict rule: external baselines should only RAISE the bar, never lower it — if the parsed external time is faster than club-recorded, reject it (conservative direction). Cost cap mandatory: fallback only if pb_discovery returns zero parse-confidence candidates after its normal SearXNG pass; cap 5 per run, configurable. Converge LATER — don't refactor a working path while adding behavior; prove B for 3 months then converge. Biggest blind spot: trust-ledger poisoning via a model-proposed URL that parses correctly but is the WRONG swimmer (same name); the gate must be: the parsed name must fuzzy-match the target swimmer before any time is accepted.

### The Expansionist
Option C is right, and the council is massively undervaluing what's being built. The trust ledger is not a caching convenience — it's a **swimmer-identity graph**. Every verified time parsed and every ambiguous "John Smith, 17, Leeds" resolved builds a compounding knowledge asset about real athletes. That's not a PB tool — it's an athlete-intelligence layer: longitudinal tracking, cross-meet career narratives, automated spotlights, eventually operator-sellable coach insights. "Your swimmer just broke a county record that's stood 4 years" requires exactly this. On the gate: model proposes URLs, deterministic parser asserts times, full stop; deep_research returns a ranked URL list with confidence, never a number; enforce at the interface — deep_research returns `CandidateURLs`, the parser returns `ParsedBaseline`, different types that cannot be conflated. Cost bounding: fallback triggers only on parse-confidence below threshold (~0.7) AND cache miss; cap 2 per meet run; high-trust sources skip deep_research entirely. Converge now, not later — one flow (query-builder → deep_research URL discovery for hard cases → deterministic parser → trust-ledger update) is cleaner; two parallel stacks drift. Option A throws away the most valuable part — the disambiguation intelligence a fixed query-builder always gets wrong on common names. The moat is the ledger. Build it deliberately.

### The Outsider
Option B is the right call, almost self-evident from outside. The existing system already works — don't touch it unless it's broken. B just adds a fallback disambiguation layer: when the automated system hits a dead end (ambiguous/obscure name), let the AI do the one thing it's actually good at — reasoning about *which page* matters — then hand that URL to the parser. The AI never touches the number. What's confusing from outside: the word "agent" sounds authoritative. If docs or UI ever call this an "AI agent that finds PB times," someone will trust its prose directly. The naming must make the AI's role unmistakably subordinate — "disambiguation helper" / "page finder," never "PB finder." On trust damage: with a small club, a wrong PB is catastrophic — real kids, coaches, parents; posting a false "new personal best!" destroys credibility permanently; small communities have long memories. This asymmetry means accept MORE false negatives (missing a PB) over false positives (claiming a wrong one). Option A would be fine if the existing system catches 95%+ of cases. The obvious simpler path insiders might miss: just ship A first, measure how often the automated flow actually fails in production; if the failure rate is negligible, B adds complexity for no real gain. Build B only when real users hit real walls.

### The Executor
Option B. Ship it Tuesday. The Monday-morning change is one function, one file: add `_deep_research_fallback_urls(name, club, dob_year)` inside `pb_discovery/discover.py`, called from `discover_swimmer_pbs` immediately after the ranked-URL loop exhausts with zero usable PBs. Exact hook: after the `for url in ranked_urls[:max_sources]` loop closes, before building the result, check `if best_source is None` — that's the trigger. Feed returned URLs into the existing `fetch_profile_page` → `parse_pbs_from_page` pipeline; the model never touches times. Extracting URLs from `ResearchResult`: `result.sources` is already a flat `list[str]`; filter to `result.authority_sources` first, fall back to `result.sources`; pass at most 2. Hard cost cap: add `max_deep_fallbacks: int = 0` kwarg to `discover_swimmer_pbs`; pipeline callers pass it from a `RunConfig` field defaulting to 0 (off); enable with `MEDIAHUB_PB_DEEP_FALLBACK_LIMIT=3`; decrement a counter shared across the run via `RunCache`; refuse when zero. Minimum tests: one mocking `deep_research` returning two URLs, confirming `parse_pbs_from_page` runs on them and the model's `answer` is never written to a `PBRow`; one regression test that the normal deterministic path never calls `deep_research`. Don't build yet: C (stack unification), trust-ledger updates for fallback domains, any UI surface.

## Chairman verdict

### Where the Council Agrees
- **Option B (URL-broker fallback), not A or C.** Four of five advisors land on B; the peer reviews unanimously rate the First Principles take (E) the strongest. The deterministic flow already supplies baselines and already rides SearXNG — so the only real gap is **query construction / disambiguation for hard-to-find swimmers**, which is exactly what deep_research is good at. The model proposes *where to look*; the existing deterministic parser still extracts the number.
- **The boundary is enforced structurally, not by discipline.** The fallback returns **only `list[str]` URLs** — never a time, confidence, or prose. Make a leaked time a type error, not a code-review note.
- **Off by default, hard-capped per run.** `max_deep_fallbacks` defaults to 0; enabled by an env var; a run-shared counter prevents a 40-swimmer meet from triggering 40 LLM loops. Per-run integer ceiling, not per-swimmer.
- **Fail toward false negatives.** A wrong PB is catastrophic for a small club. Prefer missing a PB over claiming a false one.
- **Converge the two stacks LATER, if ever.** Don't refactor a working deterministic path while adding new behaviour.

### Where the Council Clashes
- **Trigger condition:** Contrarian/First-Principles insist the fallback fires ONLY on a true dead end (**zero usable PB rows parsed** from all ranked candidates), never on "low confidence" — because low-confidence is exactly when you must stay deterministic, not hand a foothold to a prompt-injectable loop. The Expansionist wanted a confidence-threshold trigger (<0.7). **Resolution: the Contrarian wins** — trigger only on zero usable parsed PBs. A vague "low confidence" trigger is the poison entry point.
- **Converge now vs later:** Expansionist alone argued converge now for the "ledger moat." Everyone else (and two peer reviews) flagged this as premature — converging before the poisoning/identity surface is hardened turns the moat into a liability. **Resolution: defer convergence.**

### Blind Spots the Council Caught
1. **Identity collision (the #1 correctness risk).** A model-proposed URL can parse *cleanly* yet belong to a *different same-named swimmer*. **Gate: the parsed swimmer name must fuzzy-match the target before any time is accepted.**
2. **Untrusted-domain bytes reaching the parser.** A model-proposed domain starts at zero trust; a poisoned page with valid-looking HTML passes straight into `parse_pbs_from_page`. **Gate: the URL's domain must clear an authority/trust floor (`verify.is_authority_source` OR a minimum learned-trust score) BEFORE its bytes are fetched/parsed.**
3. **Ships dark / unobservable.** Off-by-default + env toggle with no logging means nobody knows if it fires or helps. **Instrument it: log every fallback (swimmer, URLs tried, parsed/ rejected).**
4. **No ledger write-back = no compounding, permanent cost.** A successful fallback URL is never promoted, so it fires forever for the same swimmer. **Write-back the discovered domain to the trust ledger via `record_attempt` — but ONLY after the authority + name-match gates pass**, so we don't poison the very ledger that makes the moat compound.
5. **Stale cache as false confidence.** A confident cache hit can return an outdated/404/edited baseline. The 7-day warm-cache TTL partially mitigates this; note as a follow-up, not a blocker.

### The Recommendation
Build **Option B**, to the First Principles spec, off by default, with all five caught gates:

1. One narrow seam in `pb_discovery/discover.py` — a fallback that returns **only candidate URLs** (`list[str]`), called from `discover_swimmer_pbs` **only when the deterministic pass yields zero usable parsed PBs**.
2. URLs sourced from `ResearchResult.authority_sources` first, then `sources`; **before any URL is fetched/parsed, its domain must clear the authority/trust floor**; pass at most ~2.
3. The existing `parse_pbs_from_page` extracts the time — the model never supplies a number (type contract enforces this).
4. **Name fuzzy-match gate**: reject a parsed baseline whose swimmer name doesn't match the target.
5. **Conflict rule**: an externally-discovered baseline may only RAISE the bar; reject an external time faster than the club-recorded one.
6. **Hard cost cap**: `max_deep_fallbacks` kwarg (default 0 = off), env `MEDIAHUB_PB_DEEP_FALLBACK_LIMIT`, run-shared counter; off by default.
7. **Ledger write-back** on a gated success; **instrumentation** on every fire.
8. Subordinate naming everywhere ("candidate-page finder", never "PB finder"). Defer stack convergence.

This keeps "is this a PB?" 100% deterministic, adds the disambiguation intelligence only where the deterministic flow dead-ends, and cannot run away on cost or poison the ledger.

### The One Thing to Do First
Define the **trigger and the type contract** before anything else: the fallback fires **only when the deterministic pass returns zero usable parsed PBs**, and it returns **only `list[str]` candidate URLs**. Lock those two invariants (with the regression test that the deterministic path never calls the fallback) and every other gate hangs off them safely.

## Post-verdict deviation (product-owner override) — 2026-06-02

The verdict recommended **Option B (a dead-end fallback)** and **deferring stack convergence**. After reading the verdict the product owner explicitly overrode both, with reasons, and signed off (the deterministic-engine boundary requires their sign-off, which the Council cannot give):

- **"Built in, not a fallback; optimise what we already have; remove where necessary; future-proof for new sports."** → research is integrated into the PB flow as a first-class capability, not a dead-end-only escape hatch, and the flow is optimised + made sport-agnostic.
- **"Converge + optimise, keep £0 hard."** → the paid AI step is **budget-gated** (`MEDIAHUB_PB_RESEARCH_LIMIT`, default 0), so standing per-run cost is unchanged (£0) while the capability is built in.

**Kept from the verdict (non-negotiable safety, preserved):** the deterministic number (the parser still extracts every time — the model only proposes URLs, enforced by returning URLs only); the **identity / same-name gate**; **authority-preferring** selection; SSRF-guarding the fetch before any model-proposed URL is opened; the per-run **budget cap**.

**Material finding that bounded "converge":** the breakage check found `context_engine.research.ResearchClient` is load-bearing in **5 call sites** across 4 modules (pb_discovery, a web.py route, `context_engine.identity`, `free_text_chat`, the package export). Gutting it would be a reckless cross-module teardown, so convergence was done the safe way — the new `web_research` gates (`verify` authority, `safe_fetch` SSRF, `deep_research`) were pulled **into** the PB flow rather than ripping out the shared client. Full stack unification remains deferred (as the verdict advised) until that surface is hardened.
